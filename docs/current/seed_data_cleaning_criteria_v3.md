# 채용공고 seed data 정제 기준

## 1. 문서 목적
- 채용공고를 seed data 파일 생성 목적에 맞게 일관되게 정제하기 위한 기준 문서
- 신규 공고 정제 시 본 문서를 기준으로 동일한 판단을 적용
- 목적은 완전한 구조화가 아니라, 공통 표현을 누적할 수 있는 초기 사전 구축
- 본 문서는 **공고 1건 단위 정제 기준**과 **누적 관리 기준**을 함께 정의한다

## 2. 기본 원칙
- 공고 원문에 명확히 드러난 표현만 우선 추출한다.
- 과도한 추론은 하지 않는다.
- 수집이 어려운 항목은 반드시 `none`으로 입력한다.
- 회사 고유명사, 제품명, 브랜드명은 기본적으로 review 또는 hold로 둔다.
- 조직문화, 태도, 성향 문장은 seed data 후보로 바로 넣지 않는다.
- alias는 실제 동의어만 남기고, 설명형 표현은 제거한다.
- 목적은 분석이 아니라 seed data 파일에 누적 가능한 공통 표현 축적이다.
- 가능한 한 **상위 분류 우선, 세부 기능 후순위** 원칙을 유지한다.
- 기준이 애매한 항목은 섣불리 확정하지 않고 review, hold, remove 중 하나로 관리한다.

## 3. 산업 분류 기준
산업은 공고 1건당 **1~3개 수준의 실제 업종 중심 상위 카테고리만** 남긴다.

### 허용 산업 카테고리
- 게임
- 여행
- 금융
- 콘텐츠
- 의료
- 커머스
- 교육
- 마케팅

### 제외 또는 보조 관리 대상
아래 표현은 산업 카테고리로 직접 확정하지 않고, 필요 시 보조 태그 또는 review 후보로 관리한다.
- AI
- IT
- SaaS
- MarTech
- 산업

### 통합 규칙
- 핀테크 / 가상자산 → 금융
- 헬스케어 / 의료기기 / 덴탈 → 의료
- 에듀테크 → 교육
- 이커머스 → 커머스

### 마케팅 산업 판단 기준
아래 중 하나 이상이 명확하면 `마케팅`을 산업 카테고리로 인정할 수 있다.
- 회사가 스스로를 마케팅 대행사, 퍼포먼스 마케팅 파트너, 마케팅 전문 기업 등으로 소개하는 경우
- 핵심 사업이 광고 운영, 퍼포먼스 마케팅, 마케팅 성과 개선, 마케팅 솔루션 제공에 직접 연결되는 경우
- 고객사의 마케팅 문제 해결, 광고 효율 개선, 마케팅 운영 체계 구축이 주된 사업 설명으로 제시되는 경우

다음과 같은 경우는 `마케팅` 산업으로 바로 확정하지 않는다.
- 일반 서비스 회사가 내부 마케팅 기능을 일부 수행하는 경우
- CRM, CMS, 광고 운영 도구 등이 등장하더라도 회사의 주된 업종이 마케팅으로 보이지 않는 경우
- 단순 홍보 문구 수준으로만 마케팅 관련 표현이 등장하는 경우

### 산업 분류 메모
- 산업은 “회사가 속한 업종” 중심으로만 판단한다.
- AI, SaaS, MarTech는 산업이 아니라 기술 성격 또는 솔루션 속성일 수 있으므로 산업으로 남발하지 않는다.
- 하나의 공고에 산업이 4개 이상 붙기 시작하면 과도 분류로 본다.

## 4. 도메인 분류 기준
도메인은 기능 단위가 아니라 **사업/서비스 영역 수준**으로만 정리한다.

### 허용 도메인 카테고리
- 게임
- 여행
- 금융
- 의료
- 이커머스
- 교육
- 마케팅

### 재검토 대상
- CRM

### 제외 규칙
- `none`은 결과 표기용 값이며, 허용 카테고리 목록에는 포함하지 않는다.
- CRM은 현재 단계에서는 사업/서비스 영역보다 솔루션 성격에 가까우므로 기본적으로 review 후보로 둔다.
- 기능 단위 표현은 도메인으로 승격하지 않는다.

### 통합 규칙
- UGC 플랫폼 / 게임 플랫폼 / 게임 서비스 → 게임
- 여행 서비스 / 항공 서비스 → 여행
- 디지털 자산 플랫폼 / 금융 플랫폼 → 금융
- 디지털 헬스케어 / 웨어러블 의료기기 / 덴탈 플랫폼 → 의료
- 커머스 플랫폼 / 이커머스 솔루션 → 이커머스
- 교육 서비스 / 학습 서비스 → 교육
- 마케팅 플랫폼 → 마케팅

### 마케팅 도메인 판단 기준
아래 중 하나 이상이 명확하면 `마케팅` 도메인으로 인정할 수 있다.
- 서비스 자체가 광고 운영, 퍼포먼스 분석, 마케팅 자동화, 마케팅 성과 관리와 직접 연결되는 경우
- 고객의 마케팅 성과를 높이기 위한 플랫폼, 솔루션, 운영 시스템이 핵심 서비스 영역인 경우
- 서비스 소개에서 마케팅 생태계, 캠페인 운영, 광고 성과 관리, 마케팅 인프라가 반복적으로 핵심 기능으로 제시되는 경우

다음과 같은 경우는 `마케팅` 도메인으로 바로 확정하지 않는다.
- 단순 CRM, CMS, 관리자 기능처럼 내부 운영 시스템 성격만 강한 경우
- 일반 서비스에서 마케팅 기능이 일부 포함된 경우
- 광고, 홍보, 브랜딩 문구만 있고 실제 서비스 영역이 마케팅으로 보기 어려운 경우

### 도메인 분류 메모
- 도메인은 “무슨 산업 안의 어떤 서비스 영역인가”를 보는 항목이다.
- 솔루션 성격, 운영 기능, 내부 시스템명은 도메인보다 review/hold 후보일 가능성이 높다.
- `none`은 도메인 카테고리 후보가 없을 때만 결과 표기용으로 사용한다.

## 5. 직무 분류 기준
직무는 세부 역할이 아니라 **채용 단위 직무명 중심**으로 정리한다.

### 허용 직무 카테고리
- 플랫폼 기획
- 서비스 기획
- 게임 기획
- AI 기획
- 프로덕트 기획

### alias / review 규칙
- PO / Product Owner → 기본적으로 `프로덕트 기획`의 alias 또는 review 후보로 처리한다.
- 직무가 명확히 PO로만 반복 출현하고, 프로덕트 기획과 구분 실익이 생기기 전까지는 독립 카테고리로 분리하지 않는다.

### 통합 규칙
- 웹 플랫폼 기획 / SaaS 플랫폼 기획 → 플랫폼 기획
- AI 서비스 기획 → AI 기획
- 솔루션 기획 / 제품 기획 → 프로덕트 기획 또는 서비스 기획
- 게임/플랫폼 기획 → 게임 기획 또는 플랫폼 기획

### 직무 분류 메모
- 현재 서비스 목적은 공고를 크게 묶어 보는 것이므로 직무 분류는 넓고 단순하게 유지한다.
- PO, PM, Product Owner처럼 회사별 역할 명칭 차이가 큰 표현은 독립 분류보다 alias/review가 더 안정적이다.

## 6. 툴/기술 정제 기준
- 공고 내 핵심 표현만 남긴다.
- 단순 나열형 전체 스택은 수집하지 않는다.
- 반복 활용 가능한 범용 표현만 우선 유지한다.
- 특수 프레임워크나 내부 기술명은 review 또는 hold 처리한다.
- 같은 파일에 함께 넣더라도 내부 판단 시 아래 성격을 구분해 본다.
  - 도구/제품
  - 기술 개념
  - 문서 산출물
- 표준 표현은 가능한 한 **명확한 명사형 중심**으로 작성한다.

### 우선 유지 대상 예시
- API
- PRD
- SRS
- UX/UI
- Jira
- Confluence
- Figma
- Slack
- SQL
- ERD
- Python
- AWS
- LLM
- Claude Code
- Cursor
- ChatGPT
- Gemini
- LangChain
- AIOps

### 작성 메모
- API, SQL, LLM은 기술 개념 성격이 강하다.
- Jira, Confluence, Figma, Cursor, Claude Code, ChatGPT는 도구/제품 성격이 강하다.
- PRD, SRS는 문서 산출물 성격이 강하다.
- 이 구분은 seed data 파일 안에서 필수 분리 항목은 아니지만, 중복 매칭 방지를 위한 내부 검수 기준으로 활용한다.
- AWS 외 동급 범용 인프라 기술 처리 기준 확장은 추후 진행 대상으로 남기며, 이번 버전에서는 별도 확장하지 않는다.

## 7. 역량 정제 기준
- 문장형 수행내용보다 재사용 가능한 범용 역량명으로 축약한다.
- 문화/성향/태도 문장은 제외한다.
- `협업`을 표준 표현으로 유지하고, `커뮤니케이션`은 기본적으로 alias 또는 보조 설명으로 관리한다.
- `서비스 운영`처럼 경계가 있는 항목은 현재 버전에서 강하게 확정 규칙을 늘리지 않는다.
- 경계 항목은 구현·운영 과정에서 실제 누적 사례를 보며 추후 정제 가능한 항목으로 둔다.

### 우선 유지 대상 예시
- 요구사항 분석
- 요건정의
- 문서화
- 정책 수립
- 프로젝트 관리
- 프로젝트 리딩
- 협업
- UX 설계
- UX 개선
- 데이터 분석
- 서비스 운영
- 기능 정의
- 문제 해결

### 통일 규칙
- 협업 = 표준 표현
- 커뮤니케이션 = alias 또는 원문 표현 보조 기록
- 단, 공고가 대외 커뮤니케이션 자체를 핵심 수행업무로 직접 요구하면 review 후보로 별도 기록 가능

### 역량 분류 메모
- `서비스 운영`, `정책 운영`, `운영 관리`처럼 경계가 맞닿아 있는 표현은 성급히 세분화하지 않는다.
- 현재는 상위 역량으로 느슨하게 관리하고, 누적 사례가 쌓이면 추후 분리 여부를 재검토한다.

## 8. review / hold / remove 기준

### review
- 향후 반복 출현 여부를 더 확인해야 하는 표현
- 기능 단위 표현
- 특수 기술명
- 세부 서비스 영역 표현
- 솔루션 성격은 있으나 상위 분류로 확정하기 이른 표현

### hold
- 회사 고유 서비스명 / 제품명 / 브랜드명
- 조직문화 문장
- 태도 / 성향 표현
- 회사 내부 용어

### remove
- 설명형 alias
- 원문 근거가 약한 임의 추론 표현
- 다른 항목과 중복되지만 독립 관리 실익이 없는 표현
- 현재 기준상 상위 표준 표현으로 충분히 흡수 가능한 표현

## 9. none 처리 기준
- 원문 근거가 없는 항목
- 억지 추론이 필요한 항목
- 현재 기준 카테고리로 안정적으로 분류하기 어려운 항목
- `none`은 결과 표기용 값이며, 카테고리 후보 자체로 누적 관리하지 않는다

## 11. 통합 누적본 관리 형식
여러 공고를 처리한 뒤에는 아래 다섯 그룹으로 통합 관리한다.

### 11.1 확정 후보
- 표현:
- 분류 축:
- 표준 카테고리:
- 누적 근거 표현:
- alias:
- 상태: confirmed

### 11.2 review 후보
- 표현:
- 분류 축:
- 임시 분류 후보:
- 보류 이유:
- 추가 필요한 사례 수:
- 상태: review

### 11.3 hold 후보
- 표현:
- 분류 축:
- 현재 해석:
- hold 이유:
- 상태: hold

### 11.4 remove 후보
- 표현:
- 분류 축:
- 제거 이유:
- 대체 표준 표현:
- 상태: removed

## 12. remove 기록 운영 원칙
- remove는 단순 삭제가 아니라, **왜 제거했는지와 무엇으로 대체했는지 추적하기 위한 기록**이다.
- remove로 기록하는 대표 사례
  - 설명형 alias를 실제 표준 표현으로 흡수한 경우
  - 세부 표현을 상위 표현으로 통합한 경우
  - 중복 표현을 하나의 표준 표현으로 정리한 경우
- remove 기록 시 가능하면 아래 두 항목을 함께 남긴다.
  - 제거 이유
  - 대체 표준 표현

## 13. 운영 원칙
- 공고를 많이 모으는 것보다 기준을 흔들지 않는 것이 우선이다.
- 동일 표현은 항상 동일 카테고리로 정제한다.
- 상위 분류 우선, 세부 기능 후순위 원칙을 유지한다.
- seed data 파일에는 공통 표현만 넣고, 고유명사는 별도 review/hold로 관리한다.
- 산업은 업종 중심으로, 도메인은 사업 영역 중심으로, 직무는 채용 단위 중심으로 본다.
- 표준 표현 수가 불필요하게 늘어나면 즉시 통합 가능성을 재검토한다.
- 경계 항목은 무리하게 확정하지 않고, 실제 누적 사례를 본 뒤 다음 버전에서 조정한다.
